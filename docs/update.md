# 前端升级说明（新库改造）

本文档面向前端，说明本次后端升级内容、影响范围、业务流变化与改造建议。

## 1. 你需要重点关注的变化

1. 角色模型变化
- 新增超级管理员 `role=3`
- 管理员为 `role=2`
- 用户禁用不再通过 role 判断，改为 `status=0`

2. 项目权限变化
- 只有超级管理员可以创建项目
- 创建项目必须指定负责人 `responsibleId`
- 普通管理员只能操作自己负责项目的开启/结束/二维码
- 超级管理员可以修改负责人

3. 参与记录模型变化（核心）
- 不再使用 `participant.status` 审核流
- 改为:
  - `is_valid`
  - `settlement_hours`
  - `note`
- 签到+签退后立即自动结算，不再走“项目结束后统一审核”模式

4. 新增统一申请流程（appeal）
- 志愿者提交申请:
  - 无效记录申诉
  - 有效记录时长变更
- 审核通过后会自动修正参与记录和累计时长

5. 会话相关变化
- 会话表移除 `openid/unionid` 冗余
- `accessToken` 和 `refreshToken` 都唯一
- 刷新逻辑已支持低概率冲突重试

## 2. 接口变更清单

## 2.1 新增接口

1. 志愿者发起申请
- `POST /api/appeals`

1. 志愿者查询可申请对象（支持按类型筛选）
- `GET /api/appeals/targets`

1. 志愿者查看自己发起的申请进度（支持按状态筛选）
- `GET /api/appeals/my`

2. 管理端申请列表
- `GET /api/admin/appeals`

3. 管理端审批通过
- `POST /api/admin/appeals/:appealId/approve`

4. 管理端审批拒绝
- `POST /api/admin/appeals/:appealId/reject`

5. 超级管理员查询管理员名单
- `GET /api/admin/admins`

6. 超级管理员修改项目负责人
- `POST /api/admin/projects/:projectId/responsible`

## 2.2 废弃接口

1. `GET /api/admin/projects/reviews`
2. `POST /api/admin/projects/:projectId/reviews/:userId/approve`
3. `POST /api/admin/projects/:projectId/reviews/:userId/reject`

## 2.3 字段变化

1. 参与记录
- 删除语义: `participant.status`（前端不要再依赖）
- 新增语义:
  - `is_valid`
  - `settlement_hours`
  - `note`

2. 志愿者汇总
- 删除: `reviewing_count`
- 保留:
  - `volunteer_hours`
  - `project_count`

3. 项目字段
- `created_by` -> `created_by_id`
- 新增 `responsible_id`

## 3. 新业务流（前端视角）

## 3.1 项目创建与管理

1. 超级管理员创建项目
- 填写项目基础信息 + 必填负责人
- 状态初始为 `0`

2. 项目开启
- 负责人管理员 或 超级管理员可开启
- 状态 `0 -> 1`

3. 项目结束
- 负责人管理员 或 超级管理员可结束
- 状态 `1 -> 2`
- 后端自动批处理参与记录

## 3.2 签到签退与结算

1. 管理端取码
- 仅负责人或超管可取签到/签退码

2. 志愿者扫码
- 扫码写入签到或签退时间
- 若已扫码同类型，幂等返回

3. 自动结算
- 当有签到和签退时即时结算
- 规则:
  - 实际时长转小时
  - 向上取整到 0.5
  - 不超过项目计划时长
- 仅首次结算累加志愿者总时长与项目数

4. 负责人自动参与
- 首位志愿者签到时自动给负责人签到
- 项目结束时自动给负责人签退并参与结算
- 负责人前端不应参与扫码

## 3.3 申请审批流（appeal）

1. 志愿者发起申请
- 提交参与记录 ID、期望时长、理由
- 可选指定期望审核员，不传默认负责人

2. 管理员审批
- 期望审核员或超管可审核
- 通过:
  - 参与记录置有效
  - 更新结算时长
  - 调整累计时长（事务）
- 拒绝:
  - 保持参与记录不变
  - 记录审核意见

## 4. 前端页面改造建议

## 4.1 权限控制

1. 超级管理员可见功能
- 创建项目
- 修改负责人
- 管理员列表

2. 普通管理员可见功能
- 项目列表
- 仅自己负责项目的开启/结束/取码
- 申请审批（只看到自己负责审核的）

3. 志愿者可见功能
- 扫码页面
- 我的参与记录
- 发起申请

## 4.2 列表与详情展示

1. 参与记录展示字段改为
- `is_valid`
- `settlement_hours`
- `note`

2. 移除前端对以下字段依赖
- `participant.status`
- `reviewing_count`

3. 项目卡片建议新增
- 负责人信息（按 `responsible_id`）
- 当前状态标签（0/1/2）

## 4.3 错误处理

1. 统一按 `code` 处理，不要只看 HTTP 状态码。
2. 40101 自动 refresh 一次，再重试原请求。
3. 对业务错误做友好提示:
- `学号已存在`
- `仅项目负责人或超级管理员可操作`
- `带队管理员由系统自动签到签退，无需扫码`

---

## 5. 对接检查清单（建议逐项验收）

1. 登录后 `auth/me` 能正确拿到 role 与 profile。
2. 超级管理员账号可以创建项目并指定负责人。
3. 普通管理员无法创建项目、无法改负责人。
4. 非负责人管理员无法开启/结束他人项目。
5. 志愿者扫码后能看到实时结算结果（有 `settlement_hours`）。
6. 负责人尝试扫码会收到明确错误提示。
7. 申请提交流程可用，列表可见，审批通过/拒绝生效。
8. 旧审核接口已全部移除调用。
9. 前端所有 `reviewing_count`、`participant.status` 引用已清理。

---

## 6. 推荐联调顺序

1. 鉴权与角色
- 登录 -> auth/me -> 管理端菜单权限

2. 项目流
- 超管创建项目 -> 指定负责人 -> 负责人开启

3. 签到签退
- 负责人取码 -> 志愿者扫码签到 -> 志愿者扫码签退 -> 查看自动结算

4. 结束流
- 负责人结束项目 -> 检查异常记录 note 与有效性

5. 申请流
- 志愿者发起申请 -> 管理端审批 -> 验证时长修正
